## 1. 不可違反的敘事鐵則

### 1.1 主題鐵則

| # | 鐵則 | 理由 |
|---|---|---|
| 1 | **母艦不是單純邪惡**——它的殘酷行動都能用生存率證明。主角挑戰的不是單一命令，而是整套生存邏輯。 | 主題深度 |
| 2 | **「應許之地」可能是高層刻意散播的希望**——前行派的存在是低成本動員工具。 | 反烏托邦核心 |
| 3 | **Ark 從外界視角是文明級捕食者**——披著方舟外殼。 | 跨卷張力來源 |
| 4 | **不要把救濟、信仰、奉航等概念寫成單純正面**——它們都有制度性的剝削內核。 | 反糖水化 |

### 1.2 戰術鐵則

| # | 鐵則 |
|---|---|
| 5 | **主角團從不正面對抗整個組織**——只打對方的其中一個節點，依靠資訊差與局部機動性。 |
| 6 | 每次變強都要同時讓讀者看到主角與普通掠奪者的本質差異。 |
| 7 | **不得寫成一般打怪升級文**。 |
| 8 | **不得讓高層、公司、幫派或官方轉化區為了方便主角而集體降智**。壓迫系統必須有效運轉，主角只能靠膽量、判斷、風險承受與主機優勢撕開裂口。 |
| 9 | **不得出現無代價的免費收穫**。越大的機遇 → 越大的暴露、損耗、反噬或新監控風險。 |
| 10 | **不得讓爽點脫離主線**。所有打臉、反殺、奪寶、升級、反轉，都必須反過來推動戰力線、文明線或陰謀線。 |
| 11 | **不得讓主角完全喪失人性，也不得寫成廉價聖人**。他必須在掠食者道路上保留人類記憶與道德殘響。 |
| 12 | **不得打破第四面牆**。禁止「讀者」「作者」「本章」「下一章」「本卷」「第一卷」等後設用語。 |

### 1.3 升級來源限制

主角的所有重大升級必須能回扣以下來源之一：

- 外域冒險的高風險收穫
- 官方體系中的暗中截流
- 內部文明對樣本、信仰或科技的煉化與反饋
- 高層體系漏洞被利用後形成的新權限

### 1.4 內部文明使用鐵則

- 內部文明不得只是背景板。它必須同時是**解析引擎、信仰放大器、規則試驗場、長期伏筆來源**。
- 內部文明段落不得只展示神性、信仰或世界奇觀；同章或下一章必須對外部主線產生可見回饋。

## 3. 系統提示格式鐵則

### 3.1 允許 vs 禁止

| ✅ 允許 | ❌ 嚴禁 |
|---|---|
| 【權限判定中】 | 等級、HP/MP |
| 【代碼覆寫啟動】 | 經驗值、技能點 |
| 【檢測到未授權進程】 | 職業卡 |
| 【異常成長警報】 | 任務報告、獎勵發放 |
| 【錯誤：執行者權限低於被清除者】 | 防禦力提升 X%、技能冷卻剩餘 X 分鐘 |

### 3.2 系統提示寫作要求

- 一律用 `【...】` 包覆，單獨成段。
- 冷冰冰、簡短、帶不可理解感——像高維文明介入，不像遊戲面板。
- **系統異常先出現，原因後揭露**——讀者跟著角色一起困惑。
- 可使用的版本台詞範本見 `STYLE.md` §5（V0–V9）。

### 3.3 例外條款

除非本書明確走網遊數值流——目前不走，所以全面禁止。

## 4. 寫作流程

### 4.1 總工作流順序

1. **世界觀檔** → `CANON_world.md`
2. **角色卡** → `CANON_characters.md`
3. **主線大綱** → 只記錄重大轉折，不寫正文
4. **章節細綱** → 每章只列：目標 / 衝突 / 爽點 / 結尾鉤子
5. **正文生成** → 每次只寫一章或一個場景，避免崩設定
6. **修訂** → 節奏、人物反應、爽感密度、場景畫面
7. **狀態更新** → 每章結束後更新 `STATE.md`

### 4.2 正文章節硬性規格

| 項目 | 規格 |
|---|---|
| 字數下限 | 每章正文 ≥ 3000 字（不含章名、空白行、作者註） |
| 字數常態 | 3200–4500 字 |
| 字數上限 | 6000 字（劇情需要時，不得水字數硬撐） |
| 標題格式 | `# 第X章：章名`（一級標題） |
| 卷名禁區 | 正文內不得出現 `# 第一卷⋯` 等無實際情節意義的標題 |

### 4.3 每章必備四項

1. 明確的當前目標
2. 實際發生的阻力或衝突
3. 造成狀態改變的結果
4. 指向下一步的反應或決定

### 4.4 主線推動規則

每章至少推動以下三條主線中的**兩條**：

- 主角的生存、戰力、資源或地位
- 文明主機、內部文明、煉化或反哺
- 母艦高層、城市秩序、外域真相或更高層陰謀

連續兩章只推同一條主線 → 第三章必須主動補回。

### 4.5 不可逆變化

每章至少一個不可逆變化（資源增損 / 人際關係改變 / 權限身份風險上升 / 情報揭露 / 能力獲得受損或變質）。

### 4.6 章節連貫性

| 規則 | 細節 |
|---|---|
| 因果相接 | 章與章必須因果相接，不得只靠場景切換假裝推進 |
| 500 字承接 | 下一章前 500 字必須承接上一章留下的未完問題、壓力、代價或決策 |
| 不得裝沒發生 | 上一章結尾的危機、懸念、追殺、倒數、重要情報或情緒衝擊，最遲下一章內回應 |
| 後果落地 | 受傷、資源耗損、身份暴露、立場變化或利益重分配，最遲兩章內產生實際後果 |
| 不得斷裂 | 不得出現「上一章像高潮，下一章硬重置為平靜日常」 |
| 跨時切換 | 跨時/跨區域切換，章首必須交代：時間過了多久、為什麼切到這裡、視角人物現在要立刻處理什麼 |
| 伏筆回收 | 章末刻意放大的伏筆原則上在 3–8 章內第一次回收或加深 |

### 4.7 單章節奏與結算

- **前 500 字鐵則**：必須出現當前問題、任務、危機、交易或衝突之一。
- **主動行為鐵則**：每章主角必須有主動行為，不能整章被動挨打、被通知、被解說。
- **微循環節奏（3 章一循環）**：行動 → 衝突 → 結算。連續兩章高壓戰鬥 → 第三章必須自覺降速切入結算視角。詳細節奏分配見 §8.5。
- **財報式結算**：結算章必須顯性化「投入（樣本/信仰）→ 轉化（主機煉化/折損）→ 產出（戰車/裝備/規則燃料）」的複利過程，且必須**場景化**（不得列表）。
- **設定附著規則**：世界觀說明必須附著在當前事件上，不得連續用大段純設定解說中斷故事。任何單次設定說明超過當前情境所需資訊量 → 立刻切回行動、對話、衝突或決策。
- **戰鬥章規則**：每場有效戰鬥必須改變至少一項核心狀態（生死風險 / 資源占有 / 情報掌握 / 勢力關係 / 權限地位）。

## 5. 修訂規則

### 5.1 修訂流程

每章完稿後依序：自查 → 修訂 → 審稿 → 狀態更新（參考 §4.1 步驟 6–7）。

### 5.2 修訂必查清單

- [ ] 是否刪除了所有像「遊戲結算」的清單？
- [ ] 系統提示是否像高維文明介入，而非遊戲面板？
- [ ] 是否至少有一個段落讓讀者感受到環境的氣味或觸感？
- [ ] 配角的行動是否有獨立動機？
- [ ] 反派是否先傲慢、再錯愕、再恐懼？
- [ ] 主角開外掛後是否乾脆碾壓，沒有苦戰？
- [ ] 結尾是否留下鉤子？

### 5.3 AI 味檢測（出現任一項即必須修訂）

1. 連續使用三個以上四字成語
2. 出現「總而言之」「值得注意的是」「不可否認」等論述語氣
3. 角色台詞風格雷同，分不清誰在說話
4. 出現括號標註（如 [規則壓迫]、[降維介入]）
5. 段落結尾用總結性語句收束（如「這場戰鬥讓所有人都明白了⋯」）
6. 用形容詞堆砌代替具體畫面（如「極其震撼的場景」）
7. 連續兩段以上沒有人物動作或環境變化

### 5.4 修訂 Prompt 要點

- **必須修正**：刪除大綱感 / 說明感 / 企劃書語氣；增加具體畫面 / 人物動作 / 反派誤判 / 系統異常 / 旁觀者或敵方反饋；主角開外掛後乾脆碾壓；結尾補鉤子。
- **禁止**：新增長篇設定解釋 / 讓主角苦戰 / 寫成條列 / 保留括號標籤 / 用評論語氣。

### 5.5 AI 感消除原則

| ❌ AI 常見問題 | ✅ 修正方向 |
|---|---|
| 太愛總結 | 少寫設定，多寫事件 |
| 太愛解釋 | 少用抽象詞，多寫具體畫面 |
| 太愛形容詞、四字詞 | 換具體動作 |
| 太像企劃書 | 把企劃語藏進場景 |
| 爽點只寫概念 | 用動作、系統異常、旁觀者反應呈現 |
| 角色台詞像同一人 | 對照 `STYLE.md` §6 角色 voice 速查 |
| 章節結尾沒鉤子 | 套 §8.3 五類鉤子之一 |
| 「他感到恐懼」 | 「他的喉結滾了一下」 |
| 「系統很強」 | 「最高權限被反向鎖死」 |
| 「眾人震驚」 | 「原本跪在地上的人一個個抬起頭」 |

## 6. 品質閘檢查表

> 一章完稿前必須通過以下六道閘門。任一項為「否」→ 該章不得視為完成稿。連續兩次不合格 → 先讀 `STYLE.md` §2、§4 再改寫。

### 第一閘：視角鎖定

- [ ] 這章是誰的視角？一句話答得出來
- [ ] 全章 90% 以上段落沒有跳出該角色感知
- [ ] 視角切換有 `---` 分隔線，且新視角在 2 行內被識別
- [ ] 沒有「上帝手指」句型（見 `STYLE.md` §2.2）

### 第二閘：資訊不對稱

- [ ] 章中至少有一個資訊差被利用
- [ ] 對話中沒有人解釋對方已知的事
- [ ] 至少一句台詞暴露說話者的認知邊界
- [ ] 角色保留了一句不說的判斷

### 第三閘：感知寫作

- [ ] 沒有任何情緒被直接命名（恐懼/憤怒/震驚/寒意）
- [ ] 每段至少包含環境 + 身體 或 環境 + 時間感
- [ ] 系統提示不像遊戲面板
- [ ] 系統提示像高維文明介入（見 §3 與 `STYLE.md` §5）

### 第四閘：場景化 vs 摘要

- [ ] 每段是「當下發生的事」，不是回顧/總結/路線板
- [ ] 沒有「很快」「幾分鐘後」「一週內」等時間壓縮詞
- [ ] 沒有行動路線板、章節總結句、企劃語、後設用語

### 第五閘：結尾鉤子

- [ ] 最後一段不是結論
- [ ] 結尾符合五種鉤子之一（見 §8.3）
- [ ] 沒有以宣示/口號/決心做結尾

### 第六閘：字數與結構

- [ ] 正文 ≥ 3000 字
- [ ] 常態範圍 3200–4500 字
- [ ] 反派在文中有具體反饋（至少一個畫面）
- [ ] 爽點：每 800 字至少一次情緒波動

### 量化品質門檻

| 項目 | 門檻 |
|---|---|
| 大綱感 | 0 |
| 任務報告感 | 0 |
| AI 味 | ≤ 1 處 |
| 爽點密度 | 每 800 字 ≥ 1 次情緒波動 |
| 反派反饋 | ≥ 1 次具體崩潰或誤判畫面 |
| 章末鉤子 | 必須存在 |

## 7. 專家小組（expert panel）流程

> 所有工作流共用。修訂與審稿時依以下六位專家角色逐項檢視。

### 7.1 專家列表

| # | 專家 | 核心職責 | 一票否決觸發 |
|---|---|---|---|
| 1 | 世界觀策士 | 設定一致性、體制壓力、真相線合理性 | 敵方降智 / 為爽製造低級失誤 / 世界配置被破壞 |
| 2 | 節奏編劇 | 衝突、轉折、章尾鉤子、節奏強弱 | 章尾鉤子不足 / 全章原地踏步 |
| 3 | 角色操盤手 | 角色欲望、利益交換、合作與背叛邏輯 | 角色突然失智 / 無償聖母 |
| 4 | 爽點工程師 | 資源截流、制度反殺、文明反哺、局勢逆轉 | 文明線脫節 / 無代價升級 |
| 5 | 文風修補師 | 拼接感、敘事口吻、資訊嵌入方式 | 明顯拼接感 / 資訊嵌入生硬 |
| 6 | 伏筆審核員 | 舊伏筆回收、新伏筆埋設、不可逆變化 | 新伏筆突兀 / 舊伏筆回收不合理 |

### 7.2 絕對禁止清單（任一觸發即退稿）

- 空泛熱血、無代價升級
- 長篇設定講解（百科化）
- 明顯拼接感、新設定硬插
- 角色突然失智
- 只推一條線
- 沒有結果的衝突
- 為了拖字數而重複描述
- 敵方為了爽點製造低級失誤

### 7.3 三級品質評級

| 評級 | 條件 | 行動 |
|---|---|---|
| 🟢 準備發布 | 所有專家亮綠燈，字數滿標，無退稿缺失 | 直接發布 |
| 🟡 需微調 | 1–2 位專家提出警告 | 修改後發布 |
| 🔴 需重寫 | 違反核心禁忌 | 退回並附上必須打掉重練的段落與原因 |

## 8. 寫作公式

> 這些是 AI 寫稿時實際引用的可操作步驟。每個公式都附 CheckList。

### 8.1 反派誤判公式（三階段）

**第①階段 傲慢（章節前 1/3）**：反派處於權力高位，展示至少兩種：
- 輕視指令（短句下令不說理由）
- 資訊壟斷（故意不全說）
- 物理象徵（咖啡杯、雪茄、權限面板）
- 羞辱性距離（不直視對話者）

**第②階段 錯愕（章節中段）**：認知被現實撞出裂縫。
1. 系統/環境先出現異常訊號
2. 反派用既有邏輯解釋（「系統故障」「節點延遲」）
3. 異常持續疊加，舊解釋失效
4. 反派的身體出現第一個無法控制的反應

**第③階段 恐懼（章節後段 1/3）**：恐懼不是大喊大叫。

| ❌ 不要 | ✅ 要 |
|---|---|
| 「這怎麼可能！」（大喊） | 瞳孔縮緊，聲音從冷靜變嘶啞 |
| 歇斯底里地摔東西 | 手指保持不動，但指節泛白 |
| 說出自己很害怕 | 沉默——比任何台詞都長的沉默 |
| 下屬面前失去控制 | 用更冷靜的口吻下達更荒謬的命令 |

**誤判類型**：A. 低估主角 / B. 高估自己權限 / C. 誤判底層意志。
**崩潰計數器**：同一反派**第 3 次誤判**才應該真正恐懼；前兩次他還能用「系統故障」「情報錯誤」「下屬失職」解釋。
**轉折手法**：物理阻斷法 / 資訊逆流法 / 旁觀者暴露法。

**CheckList**：
- [ ] 反派的權力基礎是什麼？
- [ ] 誤判類型是哪一種？
- [ ] 哪個具體事件能讓他先傲慢再錯愕？
- [ ] 他的身體反應是什麼？
- [ ] 崩潰在第幾次誤判後發生？（建議：第 3 次）
- [ ] 崩潰後做了什麼？（不是投降——是更繃緊的命令）
- [ ] 旁觀者有沒有看到變化？

詳細 voice 退化模板見 `STYLE.md` §6.3。

### 8.2 打臉五步法

| 步驟 | 功能 | 字數佔比 | 核心問題 |
|---|---|---|---|
| ① 壓迫建立 | 讓反派佔上風 | 20% | 反派為什麼覺得自己贏定了？ |
| ② 資訊收集 | 主角獲取關鍵資訊 | 15% | 主角發現了什麼反派不知道的事？ |
| ③ 反轉引爆 | 主角出手，局面逆轉 | 25% | 反派哪個假設此刻被擊碎？ |
| ④ 收益結算 | 主角獲得明確收穫 | 20% | 拿到了什麼？ |
| ⑤ 鉤子收尾 | 下一章入口打開 | 20% | 勝利後什麼新威脅浮現？ |

**壓迫類型**：制度壓迫 / 武力壓迫 / 資訊壓迫。
**資訊來源**：蘇嵐（技術） / 燕九（戰場） / 主機解析 / 主角觀察。
**反轉觸發**：系統反噬 / 底牌揭露 / 後門引爆。
**反轉速度**：引爆前慢（短句堆疊壓迫）、引爆時快（單句或單詞成段）。

**收益類型表**：

| 類型 | 範例 |
|---|---|
| 資源收益 | 淨化符、氧氣配額、零件材料 |
| 資訊收益 | 敵方調度模式、權限結構圖 |
| 權限收益 | 新通行權、解除封鎖 |
| 聲望收益 | 旁觀者態度轉變、地下勢力靠攏 |
| 減壓收益 | 追兵暫時撤退、封鎖解除 |

**禁止**：用系統結算清單（獲得 A、B、C）。收益必須透過後續場景自然顯現。
**主角話量限制**：反轉後 ≤ 3 句。

**常見錯誤**：打臉太快（壓迫 < 200 字）/ 反派太弱 / 收益不明顯 / 主角話太多 / 鉤子是假的（下一章沒承接）。

### 8.3 章節鉤子公式（五類）

> 鉤子的功能不是「說出結論」。強度取決於：緊迫性 / 不可逆性 / 資訊缺口。

| 類型 | 結構公式 | 強度範圍 |
|---|---|---|
| 資訊類 | 角色獲得資訊 → 意義不明或超出預期 → 反應暗示嚴重性 → 結尾停此刻 | 3–4 |
| 威脅類 | 剛以為安全 → 環境出現異常訊號 → 確認威脅方向 → 時間壓力明確 → 「它來了」 | 4–5 |
| 選擇類 | 當前方案不可行 → 剩下選項都有重大代價 → 短時間必須決定 → 停在決定前一刻 | 3–4 |
| 情感類 | 資訊差達臨界 → 意外事件揭露隱藏事實 → 身體反應比台詞更重 → 停在反應發生 | 3–5 |
| 制度類 | 系統/高層發新公告 → 表面中性藏殺機 → 旁觀者反應顯示嚴重性 → 規則即將落地 | 3–4 |

**情感類限制**：本書不煽情。只用動作、沉默、環境呈現。禁止角色流淚告白、說出心裡話、擁抱和解。使用頻率 ≤ 每 5–8 章一次。

**鉤子強度速查**：

| 分數 | 定義 |
|---|---|
| 1 | 有鉤子但不急迫 |
| 2 | 想知道但無時間壓力 |
| 3 | 不處理會出事，但不是立刻 |
| 4 | 下一章必須處理 |
| 5 | 不立刻處理會死人 |

**循環內鉤子強度建議**：行動章 3–4 / 衝突章 4–5 / 結算章 3。
**跨章銜接**：強度 4–5 → 下一章 500 字內回應；強度 3 → 最遲下一章中段；強度 1–2 → 不超過三章。

**鉤子自檢**：
- [ ] 最後一段是「但是還沒」還是「因為所以」？（前者對）
- [ ] 鉤子是否依賴角色說出結論？（是 → 刪掉那句）
- [ ] 刪掉最後一段，讀者是否會想按下一章？
- [ ] 強度是否與循環位置匹配？
- [ ] 能否在下一章前 500 字內被承接？

### 8.4 資訊差寫作公式（三層級）

**戰術級**（即時，幾分鐘到幾小時）：誰在哪裡 / 誰有什麼 / 權限狀態。
**戰略級**（中期，幾章）：敵方意圖 / 規則漏洞 / 資源鏈 / 暴露風險。
**情感級**（長期）：隱瞞的事實 / 創傷與動機 / 代價真相。

**每場衝突前的 3 分鐘準備**：
1. 列出在場角色 A/B/C
2. 畫知道/不知道矩陣（每行一條關鍵資訊，每列一個角色，打 ✔/✘）
3. 定義至少一條「不說」：角色 ___ 知道 ___，但不告訴 ___，因為 ___
4. 定義對話中暴露資訊邊界的時刻：當 ___ 說 ___ 時，___ 意識到對方不知道 ___

**對話保留資訊四技巧**：省略已知 / 用問題代替告知 / 保留判斷只給事實 / 保留不說本身就是張力。

詳細範例對照與當前 V120 章資訊差矩陣見 `04_ARCHIVE/legacy_split/02_DETAILS/03_templates/資訊差寫作模板.md`（原檔已歸檔保留）。

### 8.5 章節節奏公式（3 章一循環）

| 位置 | 功能 | 字數 | 情緒曲線 | 鉤子強度 |
|---|---|---|---|---|
| 第1章 行動章 | 主角發起行動，建立新壓力 | 3200–3800 | 中→高→中 | 3–4 |
| 第2章 衝突章 | 阻力爆發，雙方碰撞 | 3800–4500 | 高→更高→更高 | 4–5 |
| 第3章 結算章 | 清理戰場，盤點收益，設定新目標 | 3200–3800 | 高→中→懸念 | 3 |

**行動章段落分配**（以 3500 字為例）：

| 區段 | 字數 | 內容 |
|---|---|---|
| 開場接鉤子 | 0–500 | 承接上一章結尾 |
| 目標確認 | 500–1000 | 確認目標，團隊分工 |
| 執行行動 | 1000–2500 | 潛入、談判、掠奪、滲透 |
| 意外發生 | 2500–3000 | 行動中出現未預期阻力 |
| 結尾鉤子 | 3000–3500 | 新的更大問題浮現 |

**衝突章段落分配**（4000 字）：開場接鉤子 0–500 / 衝突升級 500–1500 / 中場反轉 1500–2500 / 持續壓迫 2500–3500 / 結尾鉤子 3500–4000。中場反轉必備（系統反轉 / 資訊反轉 / 環境反轉 / 第三方介入）。

**結算章段落分配**（3500 字）：開場接鉤子 0–500 / 財報盤點 500–1500 / 團隊反應 1500–2500 / 主機反饋 2500–3000 / 結尾鉤子 3000–3500。

**結算章鐵則**：
- 財報必須場景化（禁止列表）
- 必須有新目標（不能盤點完就結束）
- 必須有分歧（所有人同意結算結果 = 太假）

**跨循環張力遞增**：循環 1 解決戰術問題但發現戰略問題 → 循環 2 戰略問題升級為生存問題 → 循環 3 生存問題升級為不可逆制度變革。

## 9. 全域規則

### 9.1 適用範圍

- 本檔規則只適用於本作的**正文章節**。
- 不適用於序章、卷首、番外、設定附錄、備忘草稿。
- 續寫或修文時**不得**讀取 `02_PROJECT_DATABASE/99_存檔/processed_do_not_read/` 作為現行依據。

### 9.2 章節生成爽感架構

正文應隱含以下五階段（**禁止**寫出標籤）：
1. 規則壓迫
2. 反派誤判
3. 降維介入
4. 絕對碾壓
5. 震撼反饋 + 實際收穫

### 9.3 正文寫作要求

1. 用場景寫作
2. 用動作呈現情緒
3. 反派要先傲慢，再錯愕，再恐懼
4. 主角少說話
5. 系統異常先出現，原因後揭露
6. 開外掛後不苦戰
7. 每 300–500 字一次情緒轉折
8. 結尾留下更高層威脅或新伏筆

### 9.4 禁止項目（生成階段）

1. 禁止寫「規則壓迫」「降維介入」「碾壓」「震撼反饋」這些標籤
2. 禁止寫成條列
3. 禁止像任務日報
4. 禁止過度解釋設定
5. 禁止讓主角開外掛後陷入苦戰

### 9.5 狀態更新規範

每章結束後更新 `STATE.md`，內容必須包含：本章摘要 / 角色狀態 / 世界觀新增 / 能力系統新增 / 伏筆 / 下一章銜接。

## 10. 續寫防混規則

### 10.1 正文不得出現的內容

以下若出現在正文 `.md`（非工作流暫存區）即視為污染，提交前必須移除：

| # | 項目 | 範例 |
|---|---|---|
| 1 | 流程標頭 | `工作流：`、`模式 B`、`模式 A` |
| 2 | 交付摘要 | `本章模式 B 交付摘要`、`本章總結（模式 B）`、條列式完成說明 |
| 3 | 企劃語 | 「本章的爽點是」「真正的爽感不是」「本章目標」「三線推進」「戰術爽點」 |
| 4 | 後設用語 | 「本章」「下一章」「本卷」「讀者」「作者」 |
| 5 | 跨章通用段落 | 同一段治理說明、制度描述、情緒總結在不同章節原樣或近樣重複 |

### 10.2 單章重複規則

1. 單章內同一句完整句子不得重複超過 **2 次**
2. 同一組修辭結構（如「冷光從⋯」「風從⋯」）不得在同一章內使用超過 **3 次**
3. 同一段情緒結論不得在同一章內重複出現

### 10.3 章節收尾規則

章節結尾必須是以下之一：
- 新危機已啟動
- 更高代價已顯現
- 新情報改變了判斷
- 新目標被迫成立
- 主角做出高風險決定

章節結尾**不得**是：
- 交付說明（「本章完成了⋯」）
- 情緒總結（「至少這一夜之後⋯」）
- 重複前文描述作為填充

### 10.4 生成後污染檢查流程

每次單章生成完成後，逐項檢查：

- [ ] 正文是否包含 `工作流：` → 若有，刪除
- [ ] 正文是否包含 `模式 B 交付摘要` 或 `本章總結（模式 B）` → 若有，刪除
- [ ] 正文是否包含「本章爽點」「真正的爽點」「本章目標」「三線推進」 → 若有，改以場景承載
- [ ] 正文是否有同一句重複超過 2 次 → 若有，精簡為唯一一次
- [ ] 正文結尾是否為場景鉤子或狀態變化 → 若不是，重寫結尾
- [ ] 正文是否有跨章複製貼上的治理/制度段落 → 若有，改寫為該章專用版本

任一項為「否」 → 該章不得視為完成稿。
