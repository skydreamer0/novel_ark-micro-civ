# 章節完稿審查 (Chapter QA)

寫完一章後，用此工作流自動化跑完 `NOVEL_WRITING_RULES.md` 的完稿檢查清單。

## 執行步驟

### Step 1: 讀取待審查章節
// turbo
1. 讀取使用者指定的章節檔案（`docs/01_正文/第XX章_*.md`）。
// turbo
2. 讀取上一章的最後 500 字，用於檢查連貫性。
// turbo
3. 讀取 `NOVEL_WRITING_RULES.md` 的完稿檢查清單（§7）。

### Step 2: 字數統計
// turbo
4. 統計章節正文字數（排除章名、章號、空白行、作者註記），確認是否超過 **3000 字**。
5. 若低於 3000 字，標記為 ❌ 並建議需要擴寫的段落。

### Step 3: 結構審查（逐項檢核）
6. 對照 `NOVEL_WRITING_RULES.md` §7 清單，逐項判斷：

| #   | 檢核項                                   | 判斷方式         |
| --- | ---------------------------------------- | ---------------- |
| 1   | 正文超過 3000 字                         | Step 2 結果      |
| 2   | 前 500 字內出現明確問題/目標/危機        | 閱讀章首段       |
| 3   | 有清楚的目標、衝突、結果、反應/決定      | 全章掃描         |
| 4   | 推動三條主線中至少兩條                   | 標記推進了哪些線 |
| 5   | 產生至少一個不可逆變化                   | 找出具體變化     |
| 6   | 章尾留下下一章可承接的壓力/懸念          | 閱讀最後段       |
| 7   | 回應了上一章留下的問題/代價/決策         | 比對上一章結尾   |
| 8   | 避免純設定灌輸、空轉戰鬥、水字數         | 全章掃描         |
| 9   | 升級/收穫/反轉符合世界規則與主角核心優勢 | 比對設定文件     |
| 10  | 不否定前文角色動機或已建立的事件因果       | 比對前文章節     |
| 11  | 連續 2 章純追逐戰時，本章補一個制度反打場景 | 檢查前 2 章類型  |

7. 產出審查報告，格式為：
   - ✅ 通過項
   - ⚠️ 需注意項（附具體位置與建議）
   - ❌ 未通過項（附修改建議）

### Step 4: 伏筆檢查
// turbo
8. 讀取 `docs/00_企劃大綱/02_寫作與追蹤/01_伏筆追蹤表.md`。
9. 檢查本章是否回收了任何已埋伏筆（比對 🔴 待回收項目）。
10. 檢查本章是否埋了新伏筆（需記錄）。
11. 建議更新追蹤表。

### Step 5: 輸出總評
12. 整合所有結果，給出本章的總體評級：
    - **🟢 可發布**：全部通過
    - **🟡 需微調**：1-2 項未通過，問題不嚴重
    - **🔴 需重寫**：3 項以上未通過或字數不足
